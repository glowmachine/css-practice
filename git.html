Introduction

What is a project?
The thing we're working on, the files and subfolders related to it held in a single directory.
What is Git?
A version control system (VCS) that records a project's history, including copies of changes each step of the way that allow us to revert, combine, and collaborate with others.
What is a .git repository?
A hidden subfolder in a project's folder containing all the files that make Git go, like the recovery folder of a project's files and how they're linked together. These are aka the "local respositories."
What's a branch?
A version of a project.
What's a commit?
A version of a branch, like a save point in the project.
What are the 4 different states of a file?
  • Committed - These files are part of the current commit ("save point") of the current branch ("project version").
  • Staged - These files are ready to be added to the next commit of the current branch.
  • Edited - These files have been saved with edits on the computer but need to be restaged for the edits to be included in the next commit.
  • Untracked - These files are ignored and will be removed from the next commits.
What is the working directory?
The project folder, who content change based on the commit we're looking at.
What is branching?
Making a copy of a branch from its very beginning, all the way up to a specific commit (the "base"). This way, we can work on a feature without touching the main branch.
What is merging?
Taking changes from another branch and combining them into the branch we're currently on. This way, we can update a branch only when something on the other branch is ready.
What is rebasing?
Changing the commit that we originally branched from to another one. This way, we can update our branch with changes on the other branch without twisting merges.
What is Github?
A cloud service for remote repositories, where we can backup our local repositories. Services like this (Github, GitLab, Bitbucket, etc.) also allow us to backup our work and collaborate with others on the same projects at the same time.
What is fetching?
Downloading a remote branch to update the branch, but don't update any files by merging any changes. This allows us to see if there are any changes first without affecting our files, so we can decide which and if we want to merge later.
What is pulling?
Fetching, but merge all changes. If there are conflicts between local's edits and remote's changes, immediately start the process of resolving them one by one.
What is pushing?
Uploading from local to remote and update remote.
What is a pull request?
A request to pull changes into a branch on the remote repository, or "please pull my changes into your branch." They're similar to how we implicitly allow pulls from remote branches into our local branches.
What is a fork?
A remote repository that's a replica of a different remote repository. Different owners but same structure, they allow merging between forks (i.e., pulls and pull requests).

Network Setup

ssh-keygen -t ed25519
Generate an SSH key pair of type ed25519. This asymmetric encryption allows you to securely communicate with GitHub.
cat ~/.ssh/id_ed25519.pub
Print the public key id_ed25519.pub, if it exists.
ls ~/.ssh/id_ed25519
List the private key id_ed25519, if it exists. You don't need to know what it is, just that it should be there. Anyone who gets a hold of this has full access to anything that trust it via the public key, so keep it secret.
ssh -T -p 22 git@github.com
Test if SSH can connect to github via port 22, the default port for SSH
ssh -T -p 443 git@ssh.github.com
Test if SSH can connect to github via port 443, the default port for HTTPS (for when port 22 is blocked, common on public wifi networks)
~/.ssh/config
file for assigning default ports used to connect with hosts

Git Config

git init
initalize a local git repository in the current directory
~/.gitconfig
user-level file storing the git config settings below:
git config --global user.name name
set the default name for the user on commits to name
git config --global user.email email
set the default email for the user on commits to email
git config --global init.defaultBranch main
set the default name of new git projects to main
~/.gitignore_global
file used to store the names of files to exclude from the repository
  • (MacOS) echo .DS_Store >> ~/.gitignore_global will write .DS_Store to the file
  • (Windows) echo Thumbs.db >> ~/.gitignore_global will write Thumbs.db to the file
git config --global core.excludesfile ~/.gitignore_global
exclude ~/.gitignore_global and files listed within it from being tracked
git config --global core.editor "code --wait"
set the default text editor to VS Code
git remote add alias remoteurl
start tracking remoteurl named alias as a remote repository for the local repo
git remote remove alias
stop tracking alias as a remote repo for the local repo
git remote
list the aliases of the remote repos tracked by the local repo (add -v for --verbose to list URLs with aliases)

Variables

commit IDs
a 40-digit hexadecimal value, but can be referenced using any number of digits (usually 7) as long as they are unambiguous
HEAD
pointer to the current commit, switches the files available in the working directory
HEAD~n
pointer to the nth previous commit of the current branch
"detached head"
warning, HEAD points to a commit that isn't a branch tip, some actions are restricted from this location
branch name
an alias for the branch tip, its most recent commit
remote repository name
an alias for the remote repo's URL, does not have to match the repo's actual name
"remotealias/remotebranch"
the fetched copy of the remote alias and remote branch
"origin"
often the alias given to the remote repository
"main"
often the name of the main branch, synonymous with and previously known as the "master" branch

Branch Navigation

git switch branchname/commitid
move HEAD to branchname's tip or to a specific commitid
git switch -c branchname
create a branch called branchname and switch HEAD to it (edited files both staged and not yet staged will follow HEAD to the new branch)
git branch
show list of local branches
git branch -r
show list of remote branches (-r for --remote)
git branch --all
show list of all branches, local and remote
git branch branchname
create a branch called branchname, but do not switch to it
git branch -d branchname
delete branchname (-d for --delete)
git branch -D branchname
delete branchname even if it has unmerged changes (-D for --delete --force)
git branch -u remotealias/remotebranch
set the default remotealias and remotebranch for the current branch, -u short for --set-upstream-to=repoalias/remotebranch
git branch -u remotealias/remotebranch branchname
set the default remotealias and remotebranch for branchname
git reset commitid
move HEAD to commitid or HEAD~n and erase commit history after that point, the most up to date changes made so far are retained but unstaged (this is the default --mixed behavior)
  • git reset --soft commitid: move HEAD to commitid and staged the most up to date changes
  • git reset --hard commitid: move HEAD to commitid and discard all changes made after it
  • git reset can break repositories, do not use on shared repos unless you know what you're doing
  • if you mess up, use git reflog to find a HEAD@{n} to reset back to in the same way

Workflow

git add file
add file that has been edited to the staging area (and start tracking it if not already)
git add .
add all files to the staging area (and start tracking it if not already)
git rm --cached file
stop tracking file
git rm file
delete file
git restore file
restore file to its original commit state (undo edits)
git restore --staged file
remove file from the staging area (keep edits)
git restore --source=commitid file
restore file from commitid (edit and unstaged)
git revert commitid
make a new commit that undoes the changes made in commitid or range commitid...commitid
git commit
open text editor for commit message, save and exit to commit staged files
git commit -a
open text editor for commit message, save and exit to commit staged files after automatically staging tracked files
git commit --amend
replace the commit with a new commit (new commit ID as well)
git diff commitid0 commitid1
show difference between commitid0 and commitid1
git push
upload the current branch's new commits to the default remote repo's branch
git push remotealias remotebranch
upload the current branch's new commits to remotealias's remotebranch if they haven't already been set to default by git branch -u remotealias/remoterepo branchname,
git push --force-with-lease
update the remote branch with the local branch if it hasn't changed since the last fetched, only required why your local's history doesn't move forward from remote's history
git push --force
update the remote branch with the local branch while overwriting any changes
git fetch
download commits on the default remote repo, but don't merge anything
git fetch remotealias
download commits on remotealias, but don't merge
git fetch remotealias remotebranch
download commits on remotebranch of remotealias, but don't merge
git pull
download commits on the default remote repo and merge, similar syntax as git fetch (--merge is the default unless changed in ~/.gitconfig, add --rebase to the file to rebase instead)
git merge branchname
merge a local branchname into the current branch (merging, and merging by pulling, creates a new commit)
git merge remotealias/remotebranch
merge remotebranch of remotealias into current branch (this is the fetched copy of remotebranch, not the same as git pull that downloads the newest commit before merging)

Advanced Workflow

git add -p
interactive staging: stage chunks of changes at a time that can be committed separately, enter ? and q to quit early
git rebase remotealias/remotebranch
rebasing: if fetched, start rebasing the current branch to remotebranch of remotealias
git rebase --interactive commitid
squashing: start interactive rebase onto commitid
Select action for commits between HEAD and commitid
git cherry-pick commitid
cherry-picking: merge commitid onto the current branch

Stashes

git stash
create a snapshots of the current changes
git stash save type: description
create a stash with the message type: description
git stash list
list all stashes
git stash apply stash@{n}
apply changes from the nth stash
git stash pop stash@{n}
apply changes from the nth stash and remove it from the stash list
git stash drop stash@{n}
remove the nth stash from the stash list
git stash clear
remove all stashes

Viewing History

git status
show changes since the previous commit
git log
show commit history for the current branch, flags below can be combined:
  • --all : include all branches
  • --graph : include the commit tree
  • --oneline : limits details to one line per commit
git reflog
show history of head pointer movements made on the current repository, commit IDs ordered by most recent and can be used to revert git reset

Commit Messages

What are atomic commits?
  • The practice of creating one commit per logical change. Don't do any extra work like styling or refactoring. If it doesn't need to be touched for the task at hand, leave it for a separate commit.
  • It keeps history easy to read, review, and revert if needed.
  • Can lead to enormous commit history, but is easily managed with branching and squashing (interactive rebasing).
type(scope): subject
or
type(scope): subject
BLANK LINE
body
BLANK LINE
footer

The type should be...
  • build: updating tools and libraries
  • chore: updating dependencies
  • ci: updating automation
  • docs: updating documentation, no code
  • feat: updating features
  • fix: fixing bugs
  • perf: performance improvements
  • refactor: cleaning up code
  • revert: original commit message
  • style: formatting code
  • test: updating tests
The scope is optional, one word that specifies what the commit relates to.

The subject should...
  • be written in the imperative mood (a command or instruction)
  • not be capitalized (e.g. “add dark mode ui”)
  • not end with a period
  • be no more than 70 characters wide
The body should...
  • be written in the present tense
  • be exactly: This reverts commit hash if the commit is a revert
  • be should be no more than 100 characters wide
The footer should...
  • be used when a commit introduces changes that may break existing code
  • start with BREAKING CHANGE: and explain why the change is not backwards compatible
  • when present, the commit message type must be written as type! instead of type
