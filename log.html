March 16, 2026

I've been working on the demo site for the past two weeks now, and I'm unhappy that it's taken far longer than I expected. I'm learning a lot, there are just so many things that can conflict! For the most part, I've been sticking with relative units to accomodate responsive layouts later on, and the most difficult thing so far is dealing with width and height since they depend on their parents or containers, their display types, and property defaults. In no particular order, these are the things I've learned after overcoming some roadblock.

  • to create a grid, Use two linear gradients to draw two lines of a small square and repeat it across the viewport
  • to create a transparent grid, use a conical gradient that shades in 75% of a square and repeat it across the viewport, to avoid opacity stacking trying to achieve the same thing with linear gradients
  • multiple backgrounds can be layered in the same ruleset of an element
  • ::before and ::after are useful for grouping decorative styles or parts of styles shared by an element, requires content:""
  • color values can be stored and then combined with an alpha value using a function
  • position:relative allows it to be transformed from its position in normal flow
  • position:absolute is relative to its nearest ancestor who isn't position:static, use top/right/bottom/left as offset to move itself visually away from default
  • backwards direction of reasoning: require a parent to be display flex or grid because of what I want to do to an element, versus making a container display flex or grid because of how I plan to layout its elements (maybe do this sparingly, feels like a bandaid, plan it out first)
  • wireframe a site first, otherwise designing from desktop-first approach could require a lot of overriding even when using relative units since it's hard to select the most sensible modular layout elements
  • accommodating margins for decorations on an element should be built into the decoration's class ruleset
  • use half-transparent background-colors or dashed borders to visualize area while arranging elements, consider outline instead of border for edge cases
  • replaced elements (elements replaced with content like img, input, button etc.) can't have ::before or ::after pseudo elements
  • flex-shrink:0 doesn't mean it can't shrink, it just won't shrink while distributing space relative to other sibling flex items, it'll still shrink if the container doesn't have enough size for the items
  • for swipe animations, make the linear gradient 200% + 1% larger so that the edge pixel can't be visible even with pixel rounding
  • % values are relative to the parent's parents and does not check higher ancenstors, auto is the default but commonly doesn't apply for height because of page stretching
  • for overflowing sections, beware that flex:1 + min-height:auto (default) will allow the height of a container to be defined by its content, whereas flex:1 + min-height:0 will make it so that it still grows to fill the remaining space but won't grow to fit its content if its content is larger than the remaining space
  • for grids, fr are preferred to row height and column width percentages that rely on its containres width and height, and height is often undefined
  • for flex items, flex-basis overwrites height (for column flow) and width (for row flow)
  • flex and grid both have a min-height and min-width of auto by default, while block and inline are 0 by default and can shrink below content size

I've also learned things like transitions, animations, pointer actions, clip paths, bezier curves, drawing geometric shapes, and many more, but nothing that was as interesting as the issues above. In hindsight, I wish I had known more about UI/UX design before getting in so deep. I'm excited to get started on learning javascript next.

Feburary 22, 2026

I've always thought that being able to collaborate with a team was just as important a skill as the actual work to be done. With that in mind, git and github were the first things I decided to learn before jumping back into coding.

I started with a very basic understanding through the relevant lessons in theodinproject's foundations course. It felt that they taught me enough to swim in the shallow end of the pool, but not enough to stay afloat in the deep end -- and after learning what I have this past week, I'm pretty glad they didn't teach it thoroughly right from the start. It feels like there's so much depth that, for someone who's not very familiar with using command line, a full course would feel less like being taught how to swim across a pool and more like being taught how to swim across an ocean, when I've never even touched water.

That said, it was a real struggle only knowing the things I learned there, not confident enough in doing much else. I knew how to set up repos and push/pull files, but what if I made a mistake? How could I undo changes? If everything gets put on record, wouldn't those undos get messy quickly? What are forks? Are they the same as branches? What's a pull request? Are there push requests? How would I work with other devs?

For the past few days, I've been learning more about git and github, and found that creating an index helped me solidify my understanding of things. There were two big breakthroughs:

  • Realizing that the staging area isn't actually located anywhere. This distinction allowed me to make the mental leap of reimagining git as operating on a set of labels rather than actually moving files around, for example, a working directory, staging area, and local repository (though files do still move between local and remote.)
  • Realizing that everything revolves around snapshots, aka commits, and that they're basically the only things ever being worked on. Everything else was just nagivating to which snapshot I wanted to work with. Even referencing a branch is often times just a reference to its most recent commit. Git commands started making a lot more sense after that.
